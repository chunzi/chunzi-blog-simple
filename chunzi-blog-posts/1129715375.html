标签的大小写问题 PerlChina 会员中心引入了标签系统，对于标签很多人还是比较陌生，我也是第一次应用。对于标签 Perl 和 perl 的两个标签，是否应该合并为同一个标签呢？邮件列表上 fayland 和 qiang 就此展开讨论。我在这里说一下我的看法。 开始的时候，我和 qiang 一样，认为标签不应该区分大小写。于是 RJ 帮助我测试的时候，已开始提供了标签 oracle 。后来他觉得不好，想改成 Oracle ，可提交后依然是小写的 oracle 。因为我用了 find_or_create() 对于 Oracle 找到了不区分大小写的 oracle 于是什么都没有改变。显示出来的依然是小写的版本。于是我意识到，标签必须是区分大小写的。于是修改了数据库字段为 binary 。于是， 出现了两个 oracle 版本。 首先，我们不能武断的将所有的标签都强制变为大写或者小写。比如 Rails 指 Ruby 的 MVC 框架，rails 指铁路。类似的问题还有英文的单复数表现形式（Rail, Rails），以及中文和英文混杂（上海，Shanghai）。 所以，对于同一个事物，会有许多形式相近，甚至形式完全不同的关键字来描述。说其中任何一个比其他更优越是不恰当的。这种多样散漫的特性恰恰是标签的魅力所在。否则不如直接给定一个类型树，让人们往里边跳。 杂散的标签确实不太好，容易分散注意力，减弱聚合带来的效应。我们可以采取两种措施来改变这种情况：
  • 按使用率提示最佳标签。
  • 允许报告类似标签，或者直接合并标签
第一种我已经实现了。使用 perl 的有 2 人，使用 Perl 的有 7 人，那么当甲提交 perl 后，会提示有一个更好的（接受并使用人数最多的）标签 Perl 可以使用，此时甲只需一次点击，即可将原来的 perl 变更为 Perl。隐式合并。当然这种方法比较被动，因为我希望尊重标注者的选择，同时这种方法也可能不够合理，用得最多的可能是不恰当的形式。 第二种策略比较复杂，尚未实现。对于 shanghai 和 上海 两个标签，只能通过人为的辨别并合并。管理员不可能总是关注并思考这些问题，所以报告系统和通知系统还是必要的。甲发现 shanghai ，觉得应该改为 上海，于是点击报告，那些标注 shanghai 的人将收到一封邮件，询问将 shanghai 改为 上海 是否合理。标注人可以通过邮件中的一个连接直接合并，当然也可以什么都不做。 说到这里，大小写的问题已经不是主要的问题了。另外，在标签上，还有很多可以衍生和挖掘，比如和外部的标签系统结合起来可以做不少的事情。
